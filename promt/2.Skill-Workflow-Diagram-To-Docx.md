## 📂 2. Các Bước Thực Hiện Chi Tiết

### 📍 Bước 1: Phân tích & Trích xuất Dữ liệu từ Draw.io XML

Sơ đồ vẽ trên phần mềm Draw.io được lưu dưới cấu trúc XML nén hoặc không nén chứa mô hình dữ liệu đồ họa của các nút bấm, đường liên kết và phân làn.

1. **Đọc tệp tin vật lý:**
   - Xác định vị trí file sơ đồ nguồn (ví dụ: `HienKeDocs/diagram/hien_ke_workflows.drawio`).
   - Đọc nội dung XML bằng thư viện xử lý tệp tin hoặc phân tích trực tiếp thông qua script Node.js.
2. **Xác định các Tab (Trang) hoạt động:**
   - Một file Draw.io có thể có nhiều trang. Cần đọc thuộc tính `name` và `id` trong thẻ `<diagram>` để xác định đúng trang nghiệp vụ cần bóc tách (Ví dụ: `Page-1: Quy trình 1 - Hiến kế của bạn & Có thể bạn quan tâm`).
3. **Bóc tách Tác nhân (Làn thực hiện - Swimlanes):**
   - Các làn phân vai trò (Swimlane) thường là các node `mxCell` có style chứa chuỗi `swimlane` (ví dụ: `style="swimlane;startSize=20;..."`).
   - Lưu lại mã định danh `id` của từng làn này để làm gốc quy chiếu Tác nhân thực hiện cho các bước nghiệp vụ nằm bên trong làn đó (dựa vào quan hệ cha-con `parent="swimlane_id"`).
4. **Trích xuất các Bước nghiệp vụ & Đường đi:**
   - **Các bước (Nút nghiệp vụ):** Quét các `mxCell` có style dạng hình chữ nhật bo góc (`rounded=1`), khối quyết định hình thoi (`rhombus`), hoặc hình tròn khởi đầu/kết thúc (`ellipse`). Trích xuất thuộc tính `value` (tên bước nghiệp vụ) và loại bỏ sạch các thẻ định dạng HTML bên trong nhãn (ví dụ: `<div>`, `<b>`, `<br>`).
   - **Đường liên kết (Mũi tên nối):** Quét các `mxCell` có thuộc tính `edge="1"`. Đọc thuộc tính `source` và `target` để vẽ lại đúng sơ đồ tuần tự các bước từ 1 đến N, loại bỏ các bước dư thừa hoặc không thuộc luồng chính.

### 📍 Bước 2: Chuẩn hóa Thể thức Văn bản Hành chính (Nghị định 30/2020/NĐ-CP)

Tài liệu đặc tả nghiệp vụ không chỉ cần đúng nội dung mà phải tuân thủ nghiêm ngặt thể thức trình bày văn bản hành chính Việt Nam để thể hiện tính chuyên nghiệp cao nhất:

1. **Quy chuẩn Định dạng Trang (Lề trang):**
   * **Lề trái (Left margin):** **3.0 cm** (`1701` twips) - Khoảng cách bắt buộc để đóng gáy hồ sơ/sách mà không bị đè chữ.
   * **Lề phải (Right margin):** **2.0 cm** (`1134` twips).
   * **Lề trên (Top margin):** **2.0 cm** (`1134` twips).
   * **Lề dưới (Bottom margin):** **2.0 cm** (`1134` twips).
2. **Quy chuẩn Kiểu chữ & Cỡ chữ (Typography):**
   * Sử dụng thống nhất duy nhất một font chữ quốc dân: **Times New Roman** (không pha trộn các font khác).
   * **Phần thân văn bản (Body text):** Cỡ chữ **13pt** (`size: 26` trong docx) hoặc **14pt** (`size: 28`). Giãn dòng (Line spacing) từ `1.3` đến `1.5` line. Căn lề đều hai bên (**Justify**).
   * **Bảng biểu đặc tả (Tables):** Cỡ chữ trong bảng nhỏ hơn thân bài 1 tone: cỡ **12pt** (`size: 24`). Dãn dòng trong bảng đặt ở mức `1.15` đến `1.2` line để dữ liệu gọn gàng, trực quan.
3. **Quốc hiệu, Tiêu ngữ & Khung Tiêu đề:**
   * Góc trên bên phải: **CỘNG HÒA XÃ HỘI CHỦ NGHĨA VIỆT NAM** (In hoa, đậm, 12-13pt) và **Độc lập - Tự do - Hạnh phúc** (Chữ thường, đậm, 13-14pt, có đường kẻ phụ phía dưới).
   * Góc trên bên trái: Tên cơ quan chủ quản/Dự án (In hoa, chữ thường xen kẽ, đậm) và Số hiệu văn bản.
   * Tiêu đề chính: Căn giữa, viết HOA IN ĐẬM, cỡ chữ **15pt - 16pt**, dãn khoảng cách dòng phía dưới rộng rãi để tạo độ thoáng.
4. **Nhúng sơ đồ trực quan:**
   * Hình ảnh sơ đồ luồng phải được căn giữa trang hoàn hảo, căn chỉnh tỉ lệ rộng/cao cân đối với lề trang (thường độ rộng ảnh tối đa là `550` đến `580` pixel để tránh bị tràn lề phải).
   * Dưới ảnh bắt buộc phải có nhãn mô tả in nghiêng, cỡ chữ nhỏ (11-12pt), ví dụ: "*Hình 1. Sơ đồ quy trình tiếp nhận và xử lý Hiến kế hành chính*".

5. **Quy định nghiêm ngặt về ngôn ngữ thuần Việt (Không dùng tiếng Anh):**
   * > [!IMPORTANT]
     > **YÊU CẦU BẮT BUỘC:** Toàn bộ thông tin mô tả chi tiết, tiêu đề, tên bước nghiệp vụ và vai trò thực hiện trong tài liệu Word xuất ra **tuyệt đối không được phép chứa tiếng Anh** hoặc các thuật ngữ công nghệ tiếng Anh viết tắt (trừ các tên định danh hoặc viết tắt hành chính đã được luật hóa tại Việt Nam như VNeID, CCCD).
   * Mọi thuật ngữ kỹ thuật, hệ thống hoặc công nghệ thường dùng bằng tiếng Anh phải được chuyển dịch hoàn chỉnh sang ngôn ngữ hành chính tiếng Việt tương đương. Tham khảo bảng quy đổi chuẩn dưới đây:
     
     | Thuật ngữ tiếng Anh thường dùng | Thuật ngữ tiếng Việt chuẩn hành chính bắt buộc |
     | :--- | :--- |
     | `CMS` | Hệ thống quản trị nội dung / Hệ thống nghiệp vụ |
     | `Admin` / `Administrator` | Quản trị viên / Quản trị viên chuyên trách |

### 📍 Bước 3: Khắc phục triệt để lỗi OpenXML hỏng file ảnh (`image.undefined`)

> [!CAUTION]
> **LỖI NGUY HIỂM:** Khi tạo đối tượng ảnh nhúng bằng thư viện `docx` của Node.js, nếu bỏ qua hoặc khai báo sai cấu trúc ảnh, file Word xuất ra sẽ bị Microsoft Word báo lỗi "The file is corrupted and cannot be opened" do cấu trúc tệp ZIP OpenXML bị lỗi định dạng file ảnh trong thư mục `word/media/image.undefined`.

#### 🔴 Cú pháp sai dẫn đến lỗi hỏng file:
```javascript
// LỖI: Thiếu tham số xác định định dạng ảnh 'type'
const image = new ImageRun({
    data: fs.readFileSync("diagram.png"),
    transformation: { width: 550, height: 400 }
});
```
*Hệ quả:* Thư viện `docx` không xác định được kiểu MIME của tệp tin, tự động ghi đè tên tệp tin nhúng trong nhân ZIP OpenXML là `image.undefined`. Trình xem văn bản chuẩn như MS Word, WPS Office sẽ từ chối mở file ngay lập tức.

#### 🟢 Cú pháp chuẩn 100% không lỗi:
```javascript
// ĐÚNG: Khai báo tường minh thuộc tính 'type' và cấu hình kích thước chính xác
const image = new ImageRun({
    data: fs.readFileSync("diagram.png"),
    type: "png", // BẮT BUỘC: Xác định rõ định dạng ảnh để OpenXML đóng gói chính xác
    transformation: {
        width: 550, // Điều chỉnh vừa vặn khoảng trống giữa lề trái và lề phải (tối đa 560px)
        height: 380
    }
});
```

### 📍 Bước 4: Xây dựng Script Kết xuất Tài liệu Tự động

Để tạo ra một file Word đặc tả chất lượng cao, ta xây dựng script tự động hóa bằng Node.js với mô hình ES Modules (`import/export`) nhằm tối ưu hiệu năng và quản lý mã nguồn sạch sẽ. 

> [!IMPORTANT]
> **ĐƯỜNG DẪN LƯU TRỮ ĐẦU RA:** Tập tin tài liệu đặc tả nghiệp vụ `.docx` sau khi được tạo và biên dịch xong bằng script **bắt buộc phải được lưu trực tiếp vào thư mục nghiệp vụ chính thức tại:** `d:\CPLQG\CPLQG-Template\HienKeDocs\analyst\Quy_trinh_tiep_nhan_xu_ly_Hien_ke.docx` (và bản sao thay thế `..._MoiNhat.docx` khi bị khóa).

Bảng đặc tả phải tuân thủ nghiêm ngặt các quy tắc vàng về cấu trúc thông tin sau:

> [!IMPORTANT]
> **QUY TẮC TRÌNH BÀY THÔNG TIN TRONG BẢNG ĐẶC TẢ:**
> 1. **Cột Người thực hiện (Người phụ trách):** **Tên Người thực hiện phải đúng chính xác với tên tác nhân phân làn trong diagram sau khi đã lược bỏ phần nằm trong dấu ngoặc đơn.** Tuyệt đối không ghi chú thích phụ ngoài sơ đồ (Ví dụ: làn sơ đồ ghi `QUẢN TRỊ MỤC HIẾN KẾ (ADMIN)` thì cột Người thực hiện ghi đúng là `Quản trị mục Hiến kế`, không tự ý đổi thành `Quản trị viên chuyên trách` hay `Admin`).
> 2. **Cột Bước thực hiện (Tên bước):** **Loại bỏ hoàn toàn các tiền tố số thứ tự** từ nhãn sơ đồ (Ví dụ: nhãn sơ đồ ghi `1. Gửi hiến kế` thì trong bảng cột 2 chỉ ghi tên bước nghiệp vụ là `Gửi hiến kế`, tuyệt đối không ghi kèm chữ số vì cột 1 STT đã thể hiện số thứ tự).
> 3. **Cột Mô tả chi tiết nghiệp vụ (Mô tả):**
>    * **Tuyệt đối không dùng dấu gạch đầu dòng (`-` hoặc `+` hoặc `*`) đối với các dòng mô tả hành động tuần tự thông thường.** Trình bày trực tiếp dưới dạng các đoạn văn ngắn viết thường căn lề ngay ngắn (Ví dụ: `Người dân / Doanh nghiệp truy cập vào chuyên mục "Hiến kế của bạn" trên Cổng Pháp luật quốc gia.`).
>    * **Chỉ sử dụng dấu gạch ngang (`-`) khi cần phân chia các trường hợp xử lý/rẽ nhánh phức tạp** và bắt buộc phải xuống dòng rõ ràng (Ví dụ: `- Trường hợp Hợp lệ: ... \n - Trường hợp Không hợp lệ: ...`).
>    * **Danh xưng Tác nhân đồng nhất 100%:** Khi nhắc đến tác nhân trong nội dung mô tả, bắt buộc phải viết đúng chính xác danh từ ở cột Người thực hiện (Ví dụ: sử dụng đúng cụm từ `Người dân / Doanh nghiệp`, không dùng từ viết tắt hay danh xưng khác như "người dùng", "cá nhân").

Cấu trúc bảng đặc tả chuyên nghiệp gồm 4 cột chính:
1. **STT:** Số thứ tự bước (1, 2, 3...).
2. **Bước thực hiện:** Tên hành động nghiệp vụ (Không chứa tiền tố số thứ tự).
3. **Người thực hiện:** Vai trò/Tác nhân chịu trách nhiệm chính (Khớp 100% tên làn sơ đồ lược bỏ ngoặc đơn).
4. **Mô tả chi tiết:** Hướng dẫn chi tiết, đặc tả nghiệp vụ (Không dấu bullet hành động tuần tự, chỉ rẽ nhánh mới gạch đầu dòng, danh xưng đồng bộ).

> [!TIP]
> **Mẹo nâng cao trải nghiệm người dùng (UX):**
> * Viết hàm helper thông minh tự động nhận diện và gán bullet (`bullet: { level: 0 }`) **chỉ khi dòng text bắt đầu bằng dấu gạch ngang rẽ nhánh `-`**, các dòng bình thường sẽ được hiển thị dạng đoạn văn (paragraph) tự nhiên vô cùng sang trọng và thoáng đãng.
> * Thiết lập màu nền Header của bảng bằng tông màu lịch thiệp (ví dụ: Navy Blue `#1E4A8A`) với chữ trắng in đậm để tăng độ tương phản và thẩm mỹ cao cấp.

// 3. HÀM TẠO Ô TRONG BẢNG ĐẠT CHUẨN THẨM MỸ HÀNH CHÍNH
function makeCell(content, widthPercent, options = {}) {
    const { isHeader = false, align = AlignmentType.LEFT, bold = false } = options;
    let paragraphs = [];
    
    const textOptions = {
        font: "Times New Roman",
        size: 24, // Cỡ chữ 12pt đạt chuẩn bảng biểu hành chính
        bold: isHeader || bold,
        color: isHeader ? "FFFFFF" : "000000"
    };

    if (Array.isArray(content)) {
        paragraphs = content.map(text => {
            const trimmed = text.trim();
            // CHỈ gán bullet khi phân chia trường hợp xử lý (bắt đầu bằng dấu gạch ngang -)
            const isBullet = trimmed.startsWith("-");
            const cleanText = isBullet ? trimmed.substring(1).trim() : text;
            
            return new Paragraph({
                children: [new TextRun({ text: cleanText, ...textOptions })],
                alignment: align,
                bullet: isBullet ? { level: 0 } : undefined,
                spacing: { before: 60, after: 60, line: 260 }
            });
        });
    } else {
        paragraphs = [
            new Paragraph({
                children: [new TextRun({ text: String(content), ...textOptions })],
                alignment: align,
                spacing: { before: 80, after: 80, line: 280 }
            })
        ];
    }

    return new TableCell({
        children: paragraphs,
        width: { size: widthPercent, type: WidthType.PERCENTAGE },
        verticalAlign: VerticalAlign.CENTER,
        margins: { top: 120, bottom: 120, left: 160, right: 160 },
        shading: isHeader ? { fill: "1E4A8A", type: ShadingType.SOLID, color: "1E4A8A" } : undefined,
        borders: {
            top: { style: BorderStyle.SINGLE, size: 8, color: "D3D3D3" },
            bottom: { style: BorderStyle.SINGLE, size: 8, color: "D3D3D3" },
            left: { style: BorderStyle.SINGLE, size: 8, color: "D3D3D3" },
            right: { style: BorderStyle.SINGLE, size: 8, color: "D3D3D3" }
        }
    });
}

## 🛠️ 4. Kiểm Thử Cấu Trúc File & Xử Lý Sự Cố (Troubleshooting)

### 📌 Kiểm tra chất lượng đóng gói ZIP OpenXML:
Sau khi sinh file Word bằng script, để chắc chắn 100% tệp tin không bị lỗi định dạng tiềm ẩn, ta có thể kiểm thử cấu trúc nội bộ của file `.docx` (Thực chất là một file ZIP chứa các XML):

1. Thay đổi đuôi mở rộng của tệp tin vừa tạo từ `.docx` thành `.zip` (Ví dụ: `Quy_trinh.docx` -> `Quy_trinh.zip`).
2. Giải nén tệp `.zip` và truy cập sâu vào đường dẫn thư mục:
   ```bash
   Quy_trinh.zip / word / media /
   ```
3. **Kiểm tra tên file:**
   - Nếu trong thư mục `media` xuất hiện file `image1.png` (hoặc `image1.jpeg`): **ĐẠT YÊU CẦU**. Tệp tin chắc chắn sẽ mở thành công trên mọi hệ điều hành và phiên bản Microsoft Word.
   - Nếu xuất hiện file `image.undefined` hoặc `image1.undefined`: **THẤT BẠI**. Microsoft Word sẽ báo lỗi tệp tin hỏng lập tức. Bạn cần kiểm tra lại xem đã cấu hình chính xác thuộc tính `type: "png"` trong `ImageRun` chưa.

### 📌 Xử lý lỗi tập tin bị khóa (Lỗi EBUSY khi đang mở file Word):
* **Triệu chứng:** Khi chạy biên dịch script Node.js, nếu bạn đang mở tập tin `.docx` đầu ra bằng Microsoft Word trên máy tính, hệ điều hành Windows sẽ khóa quyền ghi vào file đó, gây ra lỗi `Error: EBUSY: resource busy or locked, open...`.
* **Giải pháp khắc phục tự động:** Script đã được lập trình để tự động bắt lỗi `EBUSY`. Khi phát hiện file đang bị khóa, nó sẽ in ra cảnh báo trên màn hình terminal và tự động ghi nội dung biên dịch mới ra một tệp tin bản sao có hậu tố `_MoiNhat.docx` (Ví dụ: `Quy_trinh_tiep_nhan_xu_ly_Hien_ke_MoiNhat.docx`) nằm cùng thư mục để bạn có thể xem ngay kết quả. Lần tới sau khi bạn đã đóng file Word gốc, chạy lại script sẽ ghi đè trực tiếp như bình thường.

### 📌 Làm việc trong môi trường ngoại tuyến (Offline Environment Constraints):
* **Hạn chế:** Một số máy chủ biên dịch hoặc môi trường sandbox của AI Agent hoạt động hoàn toàn offline, không có DNS kết nối ra Internet công cộng. Do đó, các tác vụ tự động kết nối qua API của Draw.io (ví dụ: `exp.draw.io` hay các headless chrome ảo để chụp màn hình từ XML) sẽ không thực hiện được.
* **Mẹo Khắc phục:** Người dùng/Developer cần tự xuất thủ công sơ đồ XML sang file ảnh định dạng PNG có độ nét cao thông qua ứng dụng Draw.io Desktop hoặc phiên bản web trước, lưu vào thư mục `HienKeDocs/diagram/hien_ke_workflows.png`. Script Node.js sau đó sẽ nhận diện ảnh vật lý này để đóng gói trực tiếp vào file Word mà không cần thực hiện thêm các thao tác render tự động.
